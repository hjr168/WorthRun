# V0.4 赛事数据运营自动化 Implementation Plan

> **For Claude:** REQUIRED SUB-SKILL: Use superpowers:executing-plans to implement this plan task-by-task.

**Goal:** 将现有手动、单批次的赛事抓取升级为可定时、可分页、可追踪、可恢复的运营流水线，并让运营人员优先处理临近比赛和缺少关键信息的候选赛事。

**Architecture:** 在现有 `apps/api` 中抽出统一的赛事源执行服务，手动触发和系统 cron 启动的一次性低内存任务共用该服务。PostgreSQL 保存调度配置、分页游标、互斥锁和每次运行记录；cron 每 15 分钟检查一次，但每次最多执行 1 个到期来源，完成后立即退出，不增加常驻 Node/PM2 进程。公开赛事仍只能由人工采纳为草稿并经过现有发布校验，不允许抓取任务自动发布。

**Tech Stack:** TypeScript、Node.js、Express、Prisma/PostgreSQL、React、Ant Design、Vitest、pnpm、cron、flock、PM2（仅现有 API）

---

## 设计决策

- 服务器可用运行内存不足 1GB，因此不增加常驻调度进程。采用系统 cron + 一次性 `eventSourceCron.ts`，任务完成即释放 V8、Prisma 和抓取解析内存；cron 不调用内部 HTTP 接口。
- cron 每次最多执行 1 个到期来源；中国田协来源默认每次 1 页、每页 20 条，硬上限为 2 页 x 20 条。分页按页顺序抓取和落库，上一页对象释放后才进入下一页，不在内存中聚合整个目录。
- 使用数据库租约锁防止手动抓取与自动抓取重叠。锁默认 30 分钟，超时后允许恢复；拿不到锁返回 `409`，不排队。
- 失败后采用 15 分钟、30 分钟、60 分钟、最长 6 小时的退避；成功后按来源配置的间隔计算下次运行时间并清零连续失败次数。
- 候选优先级由确定性规则生成，不调用模型打分。临近 30 天最高，31-90 天其次，远期再次；历史赛事最低。缺少官方入口、来源链接、日期或存在重复赛事分别保存为问题代码。
- 自动抓取只新增或刷新 `new` / `needs_review` 候选；继续跳过 `accepted`、`rejected`、`merged`，且绝不创建已发布赛事。

## 低内存运行预算

- cron 命令使用 `node --max-old-space-size=160`，将一次性任务 V8 heap 限制为 160MB；API 进程建议使用 `--max-old-space-size=256` 和 PM2 `max_memory_restart=320M`。
- cron 启动前读取 Linux `/proc/meminfo` 的 `MemAvailable`；低于 `EVENT_SOURCE_MIN_AVAILABLE_MB`（默认 256MB）时本轮跳过且不抢锁，下次 cron 再试。
- 网页正文使用流式限长读取：robots 最多 64KB、HTML 最多 400KB、抽取文本最多 30,000 字符。达到上限即取消响应流，禁止先 `response.text()` 再截断。
- 来源之间、页面之间、候选写入之间均为顺序执行。禁止对来源或分页使用 `Promise.all`；数据库分页列表默认 20 条、最大 50 条。
- 生产 `DATABASE_URL` 建议显式设置 Prisma `connection_limit=2&pool_timeout=10`，减少 API 与短时 cron 并存时的 PostgreSQL 连接内存。
- 上线验收以 Linux `/usr/bin/time -v` 的 `Maximum resident set size` 为准：一次性 cron 峰值 RSS 目标不超过 220MB，运行期间 `MemAvailable` 不低于 128MB，不能出现 OOM kill 或持续 swap 抖动。

### 后台 API

- `POST /api/admin/event-sources/:id/run`：沿用现有地址，运行记录的 `trigger=manual`；成功返回运行摘要并增加 `runId`、`startPage`、`endPage`、`pageCount`、`nextPage`。
- `GET /api/admin/event-source-runs?sourceId=&status=&page=1&pageSize=20`：返回 `{ items, total, page, pageSize }`，只允许后台角色访问。
- `PUT /api/admin/event-sources/:id`：在现有字段上接受 `scheduleEnabled`、`scheduleIntervalHours`、`pageSize`、`maxPagesPerRun`；开启调度时若 `nextRunAt` 为空则设为当前时间，暂停来源或关闭调度时清空 `nextRunAt`。
- `GET /api/admin/event-candidates`：增加 `page`、`pageSize`、`issue`、`sort=priority|newest`，响应改为 `{ items, total, page, pageSize }`；默认按 `priorityScore desc, eventDate asc, createdAt desc`。
- 不新增公开 API，不改变小程序接口。

## 验收标准

- 关闭调度的既有来源升级后不会自动运行。
- 中国田协来源能从持久化游标连续分页，单次处理量严格受配置上限控制，完成一轮后回到第 1 页。
- 手动和自动任务不能并发执行同一来源；异常退出后的过期锁可以恢复。
- 每次任务都有成功或失败运行记录，且不会记录密钥和大段网页正文。
- 失败来源自动退避，成功后恢复正常间隔。
- 自动任务没有常驻 Node 进程，低内存时会跳过，单次最多处理 1 个来源和 40 条候选。
- 真实服务器资源验收满足 cron 峰值 RSS <= 220MB、运行期间 `MemAvailable` >= 128MB，且 API 无中断。
- 后台能查看来源健康、运行历史、下一次运行时间，并按候选问题和优先级筛选。
- 自动化链路不会发布赛事；采纳仍只创建 `publishStatus=draft`。
- 现有 OpenAI、GLM、DeepSeek、页面 URL、中国田协、反馈防刷、分享和小程序流程全部通过回归验证。

## 明确不在本期范围

- 不实现任意 JSON API、RSS 或搜索引擎来源适配。
- 不增加 Redis、BullMQ、Kafka 或云任务平台。
- 不增加第二个常驻 PM2/Node 进程。
- 不做 AI 自动补全官方报名地址，不自动接受候选，不自动发布赛事。
- 不做短信、邮件、微信模板消息告警；本期通过后台来源健康和运行历史发现异常。
- 不改小程序页面和公开接口。

## 默认假设

- 生产环境继续采用单台 ECS、PostgreSQL、Nginx 和单个常驻 API PM2 进程；自动抓取只能由 cron 启动一次性 Node 任务。
- 服务器可用于新增任务的运行内存不足 1GB，因此所有默认值以低内存优先；实际服务器资源验收失败时自动调度不得上线。
- 中国田协接口仍支持 `pageNo/pageSize`；应用侧单页硬上限为 20，若真实返回缺少分页元数据，按 `totalCount/pageSize` 推导。
- 首次上线不自动开启任何来源的调度，由运营人员先手动验证后逐个开启。
- 默认调度间隔 24 小时、每页 20 条、每次 1 页、每个 cron 最多 1 个来源、单来源锁 30 分钟。
