# WorthRun V0.4.4 反馈治理与小程序体验优化计划

## Summary

在原“反馈防滥用与内测质量闭环”计划上，增加首页、赛事列表、赛事详情、反馈页和“我的”页的全局体验优化。

验收目标：

- 自动探测内容在反馈入库前被阻止，历史异常反馈可追溯治理。
- 用户提交反馈必须填写 6–500 字补充说明。
- 小程序减少重复入口，突出“判断赛事、前往官方确认、收藏”主流程。
- 用户可在本机看到最近反馈回执，但不新增匿名反馈查询接口。
- 不增加常驻进程，API RSS 不超过 220MB，维护任务峰值不超过 120MB。

## Implementation Changes

### 1. 反馈入口防护与历史治理

- 增加确定性风险分类器，识别 SQL、JNDI、脚本、路径和控制字符探测；正常中文、日期、网址和标点不得误判。
- `POST /api/feedback` 将 `userKey` 限制为 100 字符，补充说明限制为 6–500 字，并禁止内容与反馈类型完全相同。
- 命中异常规则时返回统一 HTTP 400，不返回具体规则，不保存原文、IP、请求头或设备信息。
- 新增按“北京时间日期 + 风险原因”聚合的拦截计数表，不记录用户维度数据。
- 数据治理增加 `reject_suspicious_feedback`、`reject_low_information_feedback` 和 `reject_unpublished_event_feedback`。
- 治理分类顺序固定为非法类型、异常探测、低信息、非公开赛事、完全重复；每条反馈最多进入一个动作。
- 所有治理继续使用 dry-run、预期数量保护、状态驳回和管理员操作日志，禁止物理删除。

### 2. 后台质量运营闭环

- 新增反馈摘要接口，展示待处理、可人工处理、异常探测、低信息、关联非公开赛事、重复记录及近 7/30 天拦截量。
- 反馈列表增加服务端分页、类型、状态、赛事范围和关键词筛选。
- 增加最多 50 条的批量处理预览；apply 必须携带 `updatedAt` 快照，单条冲突不影响其他记录。
- 后台页面使用紧凑统计带、风险标签和批量确认弹窗，不显示异常原文或规则表达式。
- super_admin 执行自动治理；赛事运营和内容审核角色可处理正常用户反馈。

### 3. 小程序首页与赛事列表

- 首页主标题调整为“哪场值得跑”，副标题保持“大湾区未来赛事、先判断、再去官方确认”的产品定位。
- 首页赛事分为“优先关注”“即将截止”“近期赛事”；同一赛事只能出现在一个分组，无数据的分组不渲染。
- 偏好摘要改为紧凑信息带，保留当前首选城市、首选距离和无结果降级逻辑，不扩展多值查询 API。
- 增加明确的“查看全部赛事”入口，删除重复的工具式按钮组合。
- 赛事列表补齐江门、肇庆，覆盖大湾区 9+2 城市。
- 列表显示当前结果数量和已启用筛选摘要；无结果时提供“清除筛选”操作。
- 保留现有服务端分页和触底加载，不新增排序、地图或推荐接口。

### 4. 赛事详情页信息层级

详情页顺序固定为赛事摘要、判断信息、基础信息、适合与不适合人群、确认清单、标签和合规说明。

- 删除页面上方独立的反馈卡片和分享卡片，避免与底部操作重复。
- 在赛事基础信息下增加低强调度的“信息有误？反馈纠错”入口。
- 底部栏只保留全宽“前往官方确认”、“收藏 / 取消收藏”和“分享”。
- 保留官方链接复制、收藏、分享统计和现有失败处理。
- 固定保留“AI 整理，仅供参考，报名以官方为准。”，禁止改成“立即报名”。
- 缩减详情页底部留白，确保安全区、长赛事名和三个操作在常见手机宽度下不重叠。

### 5. 反馈页与本地回执

- 反馈页根据 `eventId` 加载并展示赛事名称、日期和城市。
- 六种反馈类型改为可直接查看的选择控件，并根据类型显示填写提示。
- 补充说明必须为 6–500 字；显示字数，未满足条件时禁用提交。
- 网络失败保留已填写内容和 `requestId`；HTTP 429 显示等待时间；HTTP 400 显示可修改提示。
- 提交成功后显示完成状态，可返回赛事详情或继续浏览。
- 本地保存最近 5 条提交回执：`eventId`、赛事名称、反馈类型、提交时间和 `requestId`。
- 重试或重复提交按 `requestId` 去重，不生成多条本地记录。
- “我的”页新增“本机最近反馈”，并明确“不代表处理进度”。
- 不新增按 `userKey` 查询反馈状态的公开接口。
- “我的”页的“提交反馈”调整为“赛事纠错”，点击后进入赛事列表并提示先选择赛事。

### 6. 低内存运维保护

- 新增一次性 `feedback:maintenance` 命令，使用最大 96MB Node 堆。
- 每日清理过期反馈指纹、48 小时前的限流摘要和 90 天前的聚合拦截指标，不删除反馈正文。
- Nginx 仅针对 `/api/feedback` 设置 16KB 请求体、每 IP 每分钟 6 次、burst 3，超限返回 HTTP 429。
- 保留单个 PM2 API 进程和现有赛事源 cron，不引入 Redis、验证码、消息队列、WAF 或常驻 worker。

## Interfaces

- `POST /api/feedback`：结构不变；`userKey` 最大 100 字符，`content` 必须为 6–500 字且不能等于反馈类型；异常内容返回 HTTP 400，频率超限返回 HTTP 429。
- `GET /api/admin/feedback/summary`：返回反馈队列质量和近 7/30 天拦截指标。
- `GET /api/admin/feedback`：增加 `feedbackType`、`eventScope`、`search` 和服务端分页。
- `POST /api/admin/feedback/bulk-handle`：dry-run 返回逐条检查和快照；apply 返回 `{ handled, failed }`。
- `POST /api/admin/data-quality/cleanup`：增加异常探测、低信息和关联非公开赛事治理动作。
- 小程序本地存储新增 `worthrun_feedback_receipts`，最多保留 5 条，不上传到新的服务端接口。

## Test Plan

- 风险分类器覆盖探测变体、大小写、空白、正常中文、网址、日期和全角标点。
- 验证少于 6 字、等于反馈类型、超过 500 字和异常载荷均不能入库。
- 验证历史治理动作互斥、dry-run 稳定、预期数量冲突、重复执行和操作日志。
- 验证后台分页、筛选、最多 50 条批处理、部分成功和快照冲突。
- 验证首页赛事分组不重复，空分组隐藏，偏好降级逻辑不变。
- 验证赛事列表包含完整 9+2 城市、筛选摘要、清除筛选和触底分页。
- 验证详情页只保留一处反馈入口和一组底部操作，官方确认文案不变。
- 验证反馈表单校验、请求重试、成功状态、本地回执去重和最多 5 条限制。
- 使用微信开发者工具检查常见 iPhone 尺寸、长赛事名、安全区、键盘弹起和无内容状态。
- 执行 `pnpm test`、`pnpm typecheck`、API/admin 构建和小程序 TypeScript 检查。
- ECS 部署后验证正常反馈 HTTP 201、异常反馈 HTTP 400、频率限制 HTTP 429，且异常内容未进入数据库。
- 最终确认 PM2 仍只有一个 API 进程，API RSS 不超过 220MB，维护任务不超过 120MB。

## Assumptions

- 用户端全局优化范围限定为首页、赛事列表、赛事详情、反馈页和“我的”页。
- 偏好设置、收藏列表、跑步工具和分享卡片页面本阶段只调整入口文案，不重做内部功能。
- 不增加复杂动画、图片上传、反馈评论、消息通知或反馈处理进度查询。
- 本地反馈回执只是提交凭证，不代表管理员已查看或处理。
- 当前 11 场未来大湾区赛事作为回归基线，反馈治理不得改变公开赛事数量。
- 不覆盖或提交现有 `apps/miniapp/project.private.config.json` 用户修改。
